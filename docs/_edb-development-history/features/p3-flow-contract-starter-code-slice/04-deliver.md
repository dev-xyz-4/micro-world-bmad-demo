## 1) Scope
- Implement one bounded code slice: a **single-file Flow-Contract starter generator**.
- Convert the scaffold baseline into executable starter behavior without changing governance/routing ownership or helper workflows.
- Keep implementation reversible, architecture-neutral, and limited to one minimal code touchpoint.

### In scope
- Add one standalone script that generates starter artifacts for a new BMAD feature slice.
- Support explicit mode-aware root selection via required CLI input (no implicit routing ownership in code).
- Generate only starter planning artifacts for feature opening:
  - `01-break.md`
  - `questions.md`
- Fail safely when inputs are missing/invalid or when target files already exist.

## 2) Non-Scope
- No changes to `scripts/quality/pr-helper.sh` (no helper redesign or helper command-surface expansion).
- No governance, policy, or routing-owner document updates.
- No template-family redesign.
- No automatic creation of `02-model.md`, `03-analyze.md`, `04-deliver.md`, `README.md`, or `changelog.md`.
- No `.planning` file dependency as a required repo-tracked artifact target.

## 3) Target Files / Code Touchpoints
Exact first-slice code touchpoint:
- `scripts/quality/flow-contract-starter.mjs` (new)

Feature-local planning artifacts (already in this feature, not generated by this deliver step itself):

## 4) Implementation Steps
1. Create `scripts/quality/flow-contract-starter.mjs` as a standalone Node CLI script.
2. Implement required arguments:
   - `--feature-root` (explicit root path; caller resolves mode via `CODEX_ENTRY.md`)
   - `--feature-slug`
3. Implement optional argument:
   - `--dry-run` (print planned writes only; no file writes)
4. Validate inputs:
   - reject missing args
   - reject unsafe slug/path traversal
   - reject non-existent/invalid feature root
5. Determine target folder: `<feature-root>/<feature-slug>/`.
6. Before writing, verify that neither target file already exists.
7. If either target file exists, stop with explicit error and perform no writes.
8. If both are absent, generate exactly two files with minimal canonical starter content:
   - `<target>/01-break.md` containing:
     - `# Break — <feature-slug>`
     - `## 1) Problem Statement (one paragraph)`
     - `## 2) Goal`
     - `## 3) Non-Goals`
     - `## 4) Users / Actors (if any)`
     - `## 5) Inputs / Outputs`
     - `## 6) Constraints`
     - `## 7) Unknowns / Open Questions`
     - `## 8) Success Criteria (high level)`
     - `## 9) Classification And Versioning Note`
   - `<target>/questions.md` containing:
     - `# Questions — <feature-slug>`
     - `## Resolved`
     - `## Baseline`
     - `## Open / Blocking`
9. Print deterministic summary of planned/performed actions and exit non-zero on failure.

## 5) Acceptance Criteria
- [ ] Exactly one code file is introduced/changed for this slice: `scripts/quality/flow-contract-starter.mjs`.
- [ ] Script requires explicit `--feature-root` and `--feature-slug`; no implicit mode routing logic is embedded.
- [ ] `--dry-run` reports exact target paths and creates no files.
- [ ] Normal execution creates exactly two files (`01-break.md`, `questions.md`) only when both are absent, and those files contain the defined minimal starter scaffold headings.
- [ ] Existing target files are never overwritten.
- [ ] No changes are made to helper/governance/routing-owner/template surfaces.
